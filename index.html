UX Process

UX Engineering doesnt end with a high fidelity mockup instead it starts from there.

"I believe UX Engineering is not a task but a skill that needs to be applied at every phase of the SDLC from identifying the verticals and personas your product is targeting to building intuitive user experiences and making sure your customers are satisfied with your product." - Build the right product and, build the product right.



UX vs UI

UX UI
Includes the way a user interacts with the design, the user flow and the overall functionality of the application Includes all of the buttons, controls and design elements of the application
Focus on the people who will actually be using the design Focus on the look and feel of the design
Create wireframes and testable prototypes create finished polished designs
Work on the overall user flow of the application and how parts fit together work on individual pages/screens and interactions
Focusses more on the functionality and purpose of the design includes an artistic component, color palette, color psychology, font sizes etc…
Focus on solving ambiguous problems Create a design that is attractive
UX is data-driven mostly based on patterns, establish best practices, and personal preferences

How do I measure User Success - HEART

Happiness
Getting feedback through NPS surveys

Engagement
Using a good data analytics or tracking tool to check the frequency of users engaging with the application or product.

Adoption
can be analyzed by checking how many admins have created or added new accounts in the last 7 days, 30 days etc.

Retention

Task Success

Email Templates

What are we building?

A quick access email templating and sequencing workflow for Sales and marketing teams who spend ample amount of time in composing emails with similar templates addressing to a specific segment of people.

Why are we building?

Research says Sales Representatives spend close to two and a half hours every day on deciding what email to write or which email template to choose. To solve this issue, we will build a data driven seamless workflow wherein the user could decide which template to choose for which segment by analyzing how different templates are performing. This feature will allow users to import templates from Salesforce, build custom templates, create a sequence, Schedule, and send later, bulk send etc. Now the users could engage and target high value prospects with a crafted personalized pitch.
  • Productivity – User doesn’t need to create templates from scratch for every email or navigate to another source for copying and pasting. Everything is available right there on your email client.
  • Simplicity – Use already created templates with just few clicks.
  • Analytics – Choose the right template that is backed with required analytics showing how the templates have performed in the past

Who are we building it for?

Managers, Sales Ops, Marketing and Admins. Anyone who has an interest in how customer email communications are authored, delivered or analyzed.

UX Workflow - MVP

Users should be able to access templates built on salesforce from the sidebar and should be able to filter, search, choose and send it right away.

User Journey

I would like to know more about it
Step 1:
As a user, on an outbound email I would like to use a Salesforce template which should be available through my Zynbit sidebar and when consumed should be populated to my email compose window with the subject and ready to send mode.
Step 2:
Once my sidebar is loaded with all the recipients from the email, I choose a template.
1. The user should be able to see five templates on the dropdown, with an option to view all. *The most recent used template should be the default selected template for the next use.
Condition 1: If the selected email doesn't have any required related objects or variable tags to be selected, we populate the template on to the compose window and the user should be able to send the consumed template right away
Condition 2: If the selected template has related objects associated that need to be filled in, we load the primary recipient dropdown and related objects lookup.
Step 3:
if step 2, condition 2
Select the primary recipient from the dropdown (has all the recipients, both known and unknown listed) to whom the variable tags on the template will be applied to.
Step 4:
if step 2 , condition2
Select template → Show the related object fields that are related to the selected template (there can be multiple related object fields for a template) → Provide a lookup for the related object (For example Account, lookup accounts in SF - *not a required field, but if selected apply that to the template- like if zynbit is selected apply that account name to the template where possible ) -combo box search with typeahead. Step 5:
if step 2, condition 2
Once the primary contact from the recipients list (first contact on the recipient list on the email should be preselected by default) and the template is selected, click on apply button to publish the template to the compose window on the respected email client. Note: All the variable tags should be pre-filled where applicable.
1. Known contact - if we already have that contact on SF, we should be able to get all the info about that contact and apply it to the variable tags where applicable.
2. Unknown contact - Pre-populate will all the fields that are available with us like, the First name, last name, email etc…
3. When we copy over the template to the compose window, places where we are not able to populate the variable tags add the variable tag syntax that comes from SF like {{{First_Name}}}
Example : Hi, Sai Muchkur (when we can populate the variable tag) else Hi, {{{First_Name}}} so that the users attention would be grabbed to that part of the template and can fill in manually. - good to have - make the variable tag bold 4. Show a banner as shown in the uxpin prototype to tell the user that there might be few variable tags that should be filled in before they send the email - this is just a visual reminder kind of a thing that helps the user in not sending the email before filling in the not populated variable tags.
Step 6:
Check the email options to track the email or links before we push the template. *Condition: If there is a link inserted on the template, open the click links text field to name the link before pushing the template to the compose window. *I should be able to edit the compose window (modify the link) and then use the sidebar to refresh and name the link or make any changes to the email options.
Step 7:
If the user deletes the primary recipient on the email compose window, show a banner on the sidebar that informs the user to add back the contact or clear the template to go through a new flow again.
Step 8:
Templates from Related Lists Dropdown
Add Template as an action item on the Related List item card (Example : Contact card drill down → Related Lists - Cases → Select a case - Show templates as an action item on the card). 1. When clicked on the template icon, follow the same steps of choosing a recipient and then the template. 2. Pull up all the related objects that are required for the chosen template - there can be multiple related object fields for a template 3. Pre-fill all the required variable tags from the selected related list item. (For example - case , when chosen a template from here, since we have the case related info, pre-populate that over to the template where needed.) 4. Once the template is populated over to the compose window, if there is a variable tag that was not filled in automatically, inform the user with a banner similar to the one on the regular workflow. 5. If the user switches to a different primary recipient, show a banner saying “Variable tags only work for the recipient you have added before selecting a template. Please add back the recipient or choose a different recipient.“
